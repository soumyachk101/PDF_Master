```
                                           в””в”Җв”Җв–¶ Response { jobId }

3. PROCESS
   BullMQ в”Җв”Җв–¶ Worker в”Җв”Җв–¶ Download from S3 (input/)
                     в””в”Җв”Җв–¶ Process (qpdf/ghostscript/etc.)
                     в””в”Җв”Җв–¶ Upload to S3 (output/)
                     в””в”Җв”Җв–¶ Update PostgreSQL (job.status = completed)

4. POLL / NOTIFY
   Browser в”Җв”Җ(GET /jobs/:id)в”Җв”Җв–¶ NestJS в”Җв”Җв–¶ PostgreSQL в”Җв”Җв–¶ { status, downloadUrl }
   OR
   NestJS в”Җв”Җ(WebSocket)в”Җв”Җв–¶ Browser (real-time status)

5. DOWNLOAD
   Browser в”Җв”Җ(GET /download/:id)в”Җв”Җв–¶ NestJS в”Җв”Җв–¶ Generate Signed URL в”Җв”Җв–¶ S3 в”Җв”Җв–¶ File

6. CLEANUP (Cron)
   Scheduler в”Җв”Җв–¶ Query expired files в”Җв”Җв–¶ Delete from S3 в”Җв”Җв–¶ Update PostgreSQL
```

#### Processor Pattern (example: MergeProcessor)
```typescript
// Pseudocode for worker processing pattern
class MergeProcessor extends BaseProcessor {
  async process(job) {
    const tempDir = createTempDirectory();
    try {
      // 1. Download input files from S3
      const inputFiles = await this.downloadFiles(job.data.fileIds, tempDir);

      // 2. Execute processing
      const outputPath = path.join(tempDir, 'output.pdf');
      await execCommand('qpdf', [
        '--empty',
        '--pages', ...inputFiles.map(f => f.path), '--',
        outputPath
      ]);

      // 3. Upload result to S3
      const outputFileId = await this.uploadResult(outputPath, job.id);

      // 4. Update job status
      return { outputFileId, fileSize: getFileSize(outputPath) };
    } finally {
      // 5. Always cleanup temp files
      await removeTempDirectory(tempDir);
    }
  }
}
```

### 4.2 Auto-Scaling Triggers
| Metric | Scale Up | Scale Down |
|--------|----------|-----------|
| Queue Depth | > 20 jobs waiting | < 5 jobs for 10 min |
| Worker CPU | > 70% avg | < 30% avg for 10 min |
| API Response Time | > 2s P95 | < 500ms P95 for 10 min |

## 5. Caching Strategy

### 5.1 Cache Layers
```
Layer 1: CDN (Cloudflare)
  - Static assets (JS, CSS, images)
  - Tool page HTML (short TTL: 5 min)

Layer 2: Redis Cache
  - User session data (TTL: 15 min)
  - Rate limit counters (TTL: 1 min window)
  - Job status (TTL: 2 hours)
  - User tier/limits (TTL: 5 min)

Layer 3: Application Memory
  - Tool configurations
  - Feature flags
  - Processing tool availability
```

## 6. Failure Handling & Resilience

### 6.1 Retry Strategy
| Component | Retries | Backoff | Dead Letter |
|-----------|---------|---------|-------------|
| Job Processing | 3 | Exponential (1s, 4s, 16s) | Yes |
| S3 Upload | 3 | Linear (1s) | Fail job |
| S3 Download | 3 | Linear (1s) | Fail job |
| Virus Scan | 1 | None | Reject file |
| Email Notifications | 3 | Exponential | Log + skip |

### 6.3 Graceful Degradation
| Failure | Degradation |
|---------|-------------|
| Redis down | Disable rate limiting, use DB for queue |
| S3 down | Return error, retry later |
| Worker crash | Job auto-retries via BullMQ |
| DB down | Health check fails, return 503 |
| ClamAV down | Allow upload with warning flag |
